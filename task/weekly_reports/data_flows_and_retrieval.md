# Open-Notebook 核心对话模式数据流与检索逻辑梳理

本文档对 Open-Notebook 中的三种核心对话模式（单文档对话、普通多文档对话、Ask提问模式）在与大模型 API 交互时的数据流向和底层检索逻辑进行了详细总结。

## 1. 单文档对话模式 (Source Chat)

**适用场景**：用户在单个来源（Source）页面内，针对该特定文档进行提问。

**检索逻辑**：**无动态检索**。采用全量上下文（Full Context）加载策略。

**数据流向与处理机制**：

1. **触发流转**：用户发送消息，触发后端的 `source_chat` LangGraph 工作流。
2. **上下文构建**：
  - 系统利用 `ContextBuilder` 构建发送给大模型的背景知识。
  - 首先，强制加载该文档在入库时后台异步提取的所有**见解 (Insights)**。
  - 其次，将文档的**完整原文 (Full Text)** 追加到上下文中。
3. **截断机制**：为防止超出大模型的 Token 上限，系统设定了上下文组合总上限（原本为50000，修改为80000）以及全局字符阈值（修改为 50,000 字符）。当原文过长时，系统会优先保留 Insights，对末尾的原文进行截断。大模型并不知道自己看到的是截断后的，只会基于传入的全部字符串进行回答。
4. **API 交互**：组装好的完整上下文（Insights + 原文）加上用户的历史聊天记录，一次性打包发送给配置的 Chat 模型进行推理。此处设置了大模型输出回复的 Token 上限为 `max_tokens=8192`。
5. **前端展示**：大模型生成的文字通过 SSE (Server-Sent Events) 机制异步流式输出，在前端逐字渲染。

---

## 2. 普通多文档对话模式 (Notebook Chat)

**适用场景**：用户在笔记本（Notebook）页面内，针对该笔记本关联的多个文档和笔记集合进行自由对话。

**检索逻辑**：**无动态检索**。完全基于用户在前端界面的显式勾选内容进行拼接传输。

**数据流向与处理机制**：

1. **上下文定制（前端控制）**：用户在前端的侧边栏勾选要纳入对话的来源（Sources）和笔记（Notes），并显式选择需要引入的深度（例如选择只引入“Insights / 见解”还是“Full content / 完整原文”）。
2. **构建组装上下文**：前端发起请求，后端（`/chat/context` 路由）完全遵循前端传来的配置项，将选中的所有文档内容（对应的全文或见解）直接拼接合并在一起，形成一个巨大的背景知识字符串，**中间不进行任何针对当前提问的向量检索或语义过滤**。
3. **触发流转**：用户输入问题并发送，带着上一步已经打包好的巨量 `context` 字符串，触发后端的 `chat` LangGraph 单节点工作流。
4. **API 交互**：将这团所有文档拼装的上下文与用户的对话历史合并，一次性打包发送给 Chat 模型。此模式的字符截断逻辑与单文档完全一致，受系统统一配置的最大长度控制。模型输出回复的 Token 上限同样被设置为 `max_tokens=8192`。
5. **前端展示**：大模型的回答通过 SSE 机制以流式数据传回前端逐字渲染。

## 3. Ask 提问模式 (Ask Knowledge Base)

**适用场景**：在全局搜索页面进行跨文档提问。适用于复杂问题，系统需要像研究员一样自行规划搜索策略、综合多方信息才能得出结论。

**检索逻辑**：**Agent 任务拆解 + 分发检索 + 局部萃取综合 (Multi-step RAG 架构)**。

**数据流向与处理机制**：

1. **触发流转**：用户输入问题，触发后端的 `ask` LangGraph 工作流，这是一个包含了 3 个核心阶段的状态机。
2. **阶段一：策略规划 (`agent` 节点)**：
  - 将用户的原始问题发送给专门的**策略模型 (Strategy Model / Tools Model)**。
  - 策略模型不直接生成答案，而是执行思考，输出一段结构化的 JSON。该 JSON 包含其内部的“推理逻辑 (reasoning)”以及动态生成的 1~5 个子维度的“搜索关键词 (searches)”及针对每个关键词的提取要求。
  - 此阶段大模型生成策略的输出上限被严格控制为 `max_tokens=2000`。
3. **阶段二：检索与信息萃取 (`provide_answer` 节点 - 并发执行)**：
  - 针对策略模型生成的每一个搜索关键词，系统会在**全局知识库**中发起并发检索。
  - 检索出该词对应的 Top 文本块后，系统将这些文本块连同对应的“提取要求”，发送给专门的**应答模型 (Answer Model)**。应答模型充当信息筛选器，提炼出对应维度的局部答案。此阶段的信息萃取输出上限同样为 `max_tokens=2000`。
4. **阶段三：最终融合生成 (`write_final_answer` 节点)**：
  - 系统将阶段二中各个并发任务提炼出的所有子答案收集起来。
  - 将原始问题和所有子答案拼装成一个包含多角度论证的 Prompt，发送给**最终答复模型 (Final Answer Model)**。
  - 最终模型负责统筹所有零散的提取信息，生成结构严谨、逻辑连贯的最终答复。此阶段最后长文生成的输出上限设定为 `max_tokens=2000`。
5. **前端展示**：此模式通过双轨流式传输：阶段一策略模型的“思考过程”会作为 Token 流被捕捉并展示在前端的折叠面板中；阶段三最终模型的长文回答也会同步流式展示在下方。
