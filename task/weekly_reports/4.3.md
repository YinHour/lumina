# Open-Notebook 项目探索与改进总结 (3.28 - 4.3)

在完成了前期的基础架构梳理、解析引擎升级（Docling）之后，本阶段的工作重心全面转向了**系统并发性能优化、上下文容量突破、全链路流式交互（Streaming）重构，以及前端业务状态的实时化**。以下为按逻辑递进梳理的近 5 天工作量总结：

## Day 1: 模型适配优化与前端路由升级兼容

**核心目标：解决特定大模型接口限制与框架版本升级带来的阻断问题**

- **嵌入模型并发与限流改造**：针对用户配置的 `text-embedding-v4` 模型仅支持最大 batch size 为 10 的严格限制，深研底层向量化（`embedding.py`）逻辑。将系统的 `EMBEDDING_BATCH_SIZE` 强行修改为 10，并引入了基于 `asyncio.sleep(0.5)` 的多步异步传输缓冲机制，成功规避了 API 的并发限制与报错。
- **Next.js 15+ 动态路由修复**：排查并修复了前端页面（如 `notebooks/[id]/page.tsx` 和 `sources/[id]/page.tsx`）因 Next.js 15+ 版本更新带来的 `params are being enumerated as Promise` 报错。通过引入 `React.use()` 对路由参数进行异步解包，保障了核心业务页面的正常渲染。

## Day 2: 上下文构建器 (ContextBuilder) 逻辑重构

**核心目标：突破系统原有信息隐匿瓶颈，确保大模型获取完整文档知识**

- **隐秘逻辑排查与溯源**：针对用户反馈的“单文档对话时，大模型只能看到洞察 (Insight) 却看不到完整原文”的问题，深入溯源底层 Domain 层代码逻辑，证实了系统在短上下文模式下（`get_context("short")`）默认隐式丢弃 `full_text` 的行为。
- **构建器机制重写**：强行重构了 `ContextBuilder` 的参数透传逻辑，废弃了原有的强制 `insight` 过滤级别。现在不仅加载大模型生成的洞察，更将整个文档的 `raw_text`（原文）一并喂给大模型。
- **极限容量放宽**：匹配用户使用的 128k 超大上下文模型能力，将原文字符截断阈值大幅放宽至 50,000 字符，极大地提升了单文档深度问答的精度与依据的完整性。

## Day 3: 单/多文档对话的全双工流式输出 (SSE) 改造 (后端)

**核心目标：彻底告别“转圈等待”，实现逐字打字的极速交互体验**

- **LangGraph 异步死锁破除**：深挖底层执行图（`source_chat.py` 与 `chat.py`），发现原本使用 `model.invoke` 带来的同步阻塞（Event Loop Blocking）和线程池嵌套死锁。将其全面重构为 `await model.ainvoke` 与全异步的节点函数，打通了 LangChain 到 FastAPI 的异步通道。
- **状态机异步化替换**：由于原始的同步 `SqliteSaver` 阻碍了异步事件流的生成，将其全面替换为基于 `aiosqlite` 的 `AsyncSqliteSaver`，保障了断点记忆功能的并发安全。
- **代理缓冲层击穿**：引入标准 Server-Sent Events 流（`text/event-stream`），并在 FastAPI 响应中强行打入 `X-Accel-Buffering: no` 标头，成功击穿了 Next.js 底层代理与浏览器的静默超时缓冲机制。

## Day 4: 全局流式响应前端对接与核心状态实时更新

**核心目标：完成流式基建的最后一公里，并赋予 UI 实时响应能力**

- **前端流式解析引擎重构**：在 `useSourceChat` 和 `useNotebookChat` 钩子中重构 `fetch` 与 `ReadableStream` 逻辑，弃用会造成缓冲的 `apiClient`。实现了对 SSE 数据块 (chunks) 的缓冲拼接、不完整行的防撕裂处理、事件类型鉴别，以及打字机 UI 状态的渐进式刷新。
- **来源列表状态实时化**：针对“所有来源”页面新建后不刷新的 Bug，设计并落地了一套混合更新机制：
  - **事件驱动刷新**：在新建、更新、删除操作成功后广播自定义浏览器事件 (`sourcesUpdated`)，实现列表级的秒级重载。
  - **智能静默轮询**：引入 5 秒间隔的后台轮询 (polling) 机制，在不打断用户无限滚动 (Infinite Scroll) 体验的前提下，利用 ID 匹配智能合并、覆盖文档的 `processing` 状态与 `insights_count` 最新处理进度。

## Day 5: 深度 Ask 模式流式攻坚与多模态实验回滚

**核心目标：解决深度推理模型的兼容性卡死，完善系统多模态解析边界**

- **DeepSeek-R1 推理模型流式兼容**：排除了 Ask 模式请求长期卡死的隐蔽原因——DeepSeek-R1 等深度推理模型不兼容强约束的 JSON Structured Output API。通过剥离结构化类型约束，彻底解决了 API 挂起问题。
- **底层 Token 级深度流式捕获**：在 Ask 模式中启用 LangGraph 的 `astream_events(version="v2")`，取代了原有的按节点颗粒度返回，实现了对模型深层思考过程（`<think>`）的精准捕获。同步扩展了前端的事件解析器，对新加入的 `strategy_reasoning_chunk` 实现了保留换行符 (`whitespace-pre-wrap`) 的高保真打字机渲染。
- **多模态识别功能剥离**：响应业务需求，清理并移除了此前在 Docling 解析层中试水的“图片单独提取并调用 VLM 大模型二次描述”的功能，系统恢复为轻量级的纯文本极速解析模式。
- **架构演进调研**：详尽梳理并解答了 Ask 模式内部的查询分解与向量检索机制，为未来引入 SurrealDB 知识图谱检索（Graph RAG）提供了清晰的理论基础与架构设计建议。
