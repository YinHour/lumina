# 本周工作汇报 (4.18)

本周主要在 `feature/optim` 分支上进行了系统级的体验优化与新特性集成。为了保证系统的稳定性与解析精度，本周的工作量按照逻辑顺序拆分为以下 6 个工作日的核心任务进行推进：

## Day 1: Tavily 联网搜索能力全栈集成 (Web Search Integration)
- **底层配置与 API 扩展**：在后端的 `ContentSettings` 领域模型和数据库中新增了 `tavily_api_key` 与 `tavily_include_domains`（白名单搜索域）字段，并完善了相关的后端 API 路由。
- **大模型 Tool 绑定**：编写了 `tavily_search` 异步工具函数，并在 LangGraph 工作流（`chat.py` 与 `source_chat.py`）中通过 `enable_web_search` 参数，实现对联网检索大模型工具的动态绑定与挂载。
- **前端交互闭环**：在 Settings 页面新增了“Web Search”专属配置卡片（支持持久化与自动重置）；在聊天输入框（ChatPanel）顶部新增了带有地球图标的“联网搜索”独立开关，实现了用户意图到引擎行为的精准控制。

## Day 2： 引用规范系统与幻觉防呆设计 (Robust Citation System)
- **双轨引用体系建立**：通过极度严格的系统提示词，彻底规范了大模型的引用行为，强制要求其输出“内联超链接 `[1](URL)`”加“文末单一独立参考列表 `## 参考文献`”的标准学术格式。
- **Few-Shot 反面教材防御**：针对大模型常见的“跳号”、“不带链接”、“写出双结尾”等强迫症幻觉，在 Prompt 中直接写入了 `BAD EXAMPLES (Do NOT do this)`，通过负面样本实现了极高成功率的格式对齐。
- **本地与网络引用隔离**：重构了前端的渲染逻辑（`source-references.tsx`）。新增了针对网络链接最大编号的自动探测（`getMaxWebCitationNumber`），使得本地工作区文件（Source/Note）的编号强制从网络引用之后开始，彻底解决了“编号撞车”导致指向不明的问题。
- **前端渲染后处理（兜底）**：针对大模型偶尔在文末只输出裸链接而忘记带 `1. 2.` 编号的边缘情况，专门编写了正则表达式后处理器（`source-references-bibliography.ts`）自动补全缺失的有序列表序号，并补充了 Vitest 单元测试覆盖各种极端排版，确保了最终呈现的完美无瑕。

## Day 3. 流式输出（SSE）底层拦截与防泄漏 (Streaming & Tool Output Filtering)
- **工具输出 XML 化**：为了防止大模型在思考过程中产生“回音（Echoing）”，将 Tavily 工具的原始返回内容包裹在了严格的 `<web_search_results>` XML 标签内。
- **SSE 流式过滤拦截**：在 FastAPI 的 `stream_chat_response` 底层，增加了对 XML 标记的实时嗅探和屏蔽机制。同时过滤了历史记录接口（`get_session`）中的 Tool 执行消息，彻底解决了过去“原始搜索结果会在回答时前置泄漏并弹出在前端”的恶性 Bug。
- **依赖固化**：在 `pyproject.toml` 中显式指定并锁定了 `tavily-python` 依赖，保证了线上与本地构建的一致性。

## Day 4: MinerU 文档解析引擎全面适配与环境修复
- **引擎集成与 API 暴露**：在后端的 `ContentSettings` 领域模型和前端的设置表单中全面引入了最新的 `MinerU (magic-pdf)` 解析引擎选项。
- **底层解析工作流劫持**：在 `open_notebook/graphs/source.py` 中编写了专门的进程拦截器，针对 PDF 等复杂排版文档自动挂载 MinerU 进行处理，并支持在解析失败时优雅降级（Fallback）至 Simple 引擎。
- **环境与依赖 Bug 修复**：深入 `mineru_vl_utils` 第三方库底层，编写自动化补丁脚本（`patch_mineru.py`），彻底修复了与 `transformers` 库的 `Qwen2VLConfig` 版本不兼容导致的内存溢出及加载崩溃问题，并为下载大模型配置了国内加速节点（HF_ENDPOINT）。

## Day 5: 笔记优先级重构与检索调度优化
- **检索权重反转（Truncation Weights）**：在 `ContextBuilder` (`open_notebook/utils/context_builder.py`) 中修改了长下文截断的默认逻辑，将原本倾向于保留 Source 文本的权重反转，大幅提高 Notes（笔记）的保留优先级（`{"note": 100, "source": 75}`）。
- **强制前置排序机制**：在 Ask 模式的图搜索工作流中，加入了专门的自定义排序逻辑，强制将所有带有 `note:` 前缀的节点在搜索结果队列中置顶。
- **Context 获取模式变更**：修改 Chat 模式的行为，将拉取笔记内容的方法从之前的摘要模式 (`short`) 正式更改为全文本模式 (`long`)，确保笔记的细节无损输入给大模型。

## Day 6: 知识图谱 (KG) 数据库索引优化与查询修复
- **向量数据库结构升级**：编写并执行了第 15 号数据库迁移脚本（`15.surrealql`），为知识图谱实体表（`kg_entity`）的 `name` 和 `description` 字段补充了全新的 BM25 全文搜索索引。
- **混合检索 SQL 修复**：修复了 `graph_search` 在查询时的严重语法崩溃 Bug。重构了底层的 SurrealQL 语句，通过分离匹配占位符（`@1@` 和 `@2@`）并结合 `math::max([search::score(1), search::score(2)])`，实现了对图谱实体名称和详细描述的精准双重匹配。

## Day 7: Prompt Engineering 与特定领域知识注入 (Domain Expertise)
- **油田化学领域强化**：重构了系统的核心提示词模板（`chat/system.jinja` 及 `source_chat/system.jinja`），创造性地引入了“触发式领域专家”模式（Domain Expertise: Oilfield Chemistry & Cement Admixtures）。
- **四维分析框架**：强制大模型在涉及 AMPS、SSS 等单体及降失水剂配方时，必须采用“机理验证 (Mechanism Validation)”、“配方分析 (Formulation Analysis)”、“风险提示 (Risk Warning)”和“行动建议 (Action-Oriented Output)”的结构化专家框架进行深度推理，兼顾了通用性与垂直领域的专业深度。
